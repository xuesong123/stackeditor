这篇文档是 AI 写的，对于现状的介绍和技术选型有些价值，所以保留下来。

其中第 4 节与 StackEditor 的实现不同，大部分描述基本符合，但是对于图片的处理不同.

对于图片的处理，StackEditor 并不是立即上传，而是选择由开发者自行处理。

对于粘贴之后的文档中包含的图片，用户可能还会继续做编辑，也有可能选择直接删除，无论是保留还是删除，个人认为此时上传都不是个好选择，不如放在保存文档的时候一次性的上传。有如下原因：

第一，此时上传，如果图片比较多，就需要阻塞编辑先上传，或者不阻塞编辑而是放在任务中上传，这都会导致交互的复杂。

第二：图片上传完之后还需要将图片的URL再次替换回去，这需要提交事务，编辑器会多一次保存历史。ProseMirror 官方给了一个示例，可以做一个图片占位符，等图片上传完再写回去，这个可能会避免产生事务，但是个人认为仍然不好，需要再单独编写一套插件。

第三，如果编辑器默认实现了这些功能但是又不符合具体业务场景的要求，开发者还要自己开发，与其如此，不如在保存到服务端之前一次性上传。上传完毕之后写回去，可完美避开 ProseMirror 。



对于自动扫描 BASE64 编码的图片，StackEditor 提供了一个类：

```
ResourceHandler.scan = function(element, upload, callback) {
	...
};

ResourceHandler.scan(container, myupload, function() {
	// 提交保存
});
```



实际上，StackEditor 中很多功能都需要和 ProseMirror 交互，但是这种交互通常意味着可能产生事务，有事务就会产生操作历史，但这些功能其实属于编辑器本身的行为，不能简单的归结为文档变更。为了避免产生事务，StackEditor 中采用了一些特殊的方法，避免和 ProseMirror 交互。例如块高亮和代码高亮等功能。



富文本编辑器选型指南与自研实践 (2026 版)
----------------------------------------

1. 背景与挑战
在构建现代 Web 应用时，富文本编辑器（RTE）是交互最复杂的组件之一。我们需要在功能丰富度、开发成本、包体积和定制化能力之间做出权衡。特别是面对从 Word 复制粘贴这种高频且复杂的场景，不同方案的差异巨大。
关键约束：所有现代剪贴板操作（navigator.clipboard）均强制要求 HTTPS 安全上下文（本地 localhost 除外）。

2. 主流方案全景图

3. 为什么我们选择 ProseMirror 自研？
在经过深入调研后，我们决定基于 ProseMirror 从零构建我们的编辑器。主要基于以下考量：

**核心理由**
数据模型的绝对控制：
传统编辑器将内容存储为 HTML 字符串，导致状态管理困难。
ProseMirror 维护一个不可变的文档树（Document Tree），每一步操作都是一个 Transaction。这使得实现“撤销/重做”、“协同编辑”、“实时校验”变得异常可靠。

**极致的性能优化：**
通过细粒度的 DOM 更新策略，只重绘变化的节点。在处理长文档和复杂嵌套结构时，性能远超基于 contenteditable 简单封装的方案。

**定制化交互能力：**
我们可以轻松实现类似 Notion 的 Slash Command (/)、拖拽块（Drag & Drop Blocks） 以及内联评论，而无需与编辑器的“黑盒”逻辑斗争。

**摆脱历史包袱：**
不依赖任何特定的前端框架（React/Vue），核心逻辑纯原生 JS/TS，便于长期维护和跨框架复用。



4. 攻克的核心难点：Word 粘贴处理
  自研最大的挑战之一是处理 navigator.clipboard 和 Word 的脏数据。我们的解决方案：
  剪贴板读取策略：
  优先使用 navigator.clipboard.read() 遍历 ClipboardItem。
  检测 text/html 类型，提取 Word 生成的复杂 HTML。
  检测 image/* 类型，将图片转换为 Blob 并上传至对象存储，返回 URL 插入文档。
  数据清洗管道 (Sanitization Pipeline)：
  利用 DOMParser 解析 HTML。
  白名单过滤：仅保留标准的 <p>, <h1-h6>, <ul/ol>, <table>, <img> 等标签。
  样式剥离：正则移除所有 mso- 开头的微软私有样式，移除内联 style 中的非关键属性（如字体大小、颜色可根据需求保留或统一重置）。
  结构转换：将 Word 的复杂嵌套表格转换为 ProseMirror Schema 定义的标准 Table 节点。
  事务提交：
  将清洗后的数据结构化为 ProseMirror 的 Slice，通过 view.dispatch(tr.replaceSelection(slice)) 原子化插入。
